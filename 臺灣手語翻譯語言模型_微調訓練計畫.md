# 臺灣手語翻譯語言模型（中文 → TSL Gloss）微調訓練計畫

撰寫日期：2026-07-17
定位：本計畫對應整體專題架構「中文 → **臺灣手語翻譯 → TSL Gloss／動作腳本** → 動作生成或動作庫檢索 → …」中的語言模型環節。
原則：所有方法設計均標註可查證的出處；屬於本專案自行決定的工程選擇會明確標示「本計畫決策」。

---

## 1. 任務定義

**任務：Text-to-Gloss（T2G）翻譯** — 輸入中文句子，輸出臺灣手語 Gloss 序列（含非手部標記 NMS 欄位）。

依據：

- SignAlignLM（Findings of ACL 2025）明確定義 T2G（口語文字 → 手語 Gloss）為手語處理的標準任務之一，是 G2T（Gloss → 文字）的逆向任務。[來源 1]
- CCL24-Eval 任務 10 系統報告採用完全相同的管線：使用者輸入中文 → LLM（結合知識庫語料）翻譯為 Gloss → 手語數字人以 3D 動畫呈現手語動作、面部表情、口型。這與本專題架構一對一對應。[來源 2]
- 工研院「AI手語-虛擬氣象主播」（與公視合作，2025-03 發表）同樣以「手語文字翻譯（LLM）→ 骨架 → 影片生成」三段式架構，證明此管線在**臺灣手語**上實際可行，且預計 2025 年底於公視新聞頻道上線。[來源 5]

**輸出格式（本計畫決策，依據自有標記表既有格式）：**

Gloss 以 `/` 分隔、NMS（非手部標記）獨立欄位。例（取自本專案「台灣地名應用句子」標記表 P04）：

| 中文 | Gloss | NMS |
|---|---|---|
| 我明天要去花蓮。 | 明天/我/去/花蓮/要 | — |
| 你住在宜蘭嗎？ | 你/宜蘭/住 | 眉毛上揚（是非問句，貫穿全句） |

把 NMS 獨立成欄位的理由：Gloss 本身有「資訊丟失」的已知缺點——缺乏面部表情、身體語言等非手動通道與 3D 空間線索（CCL 論文引 Müller et al. 2023 的整理）[來源 2]；而臺灣手語的是非問句、否定、疑問正是靠非手部標記承擔語法功能（Tai & Tsay 2015；Jane Tsay 2021）[來源 7]。CCL 系統的做法是把「表情塊」與「手語塊」組成同步塊一起輸出給數字人端 [來源 2]，本計畫沿用此設計，讓下游「表情、頭部、身體同步」模組有資料可用。

## 3. 第一階段：資料建置（預估 3–4 週）

### 3.1 現有資料盤點

| 資料 | 內容 | 對語言模型的用途 |
|---|---|---|

| SLR_V2 常用語句批次（S01–S30） | 30 句，含標記表（逐 Gloss 的 start/end time） | 中文↔Gloss 平行句對 |
| 台灣地名應用句子批次（P01–P05） | 5 句，標記表含 NMS 欄位 | 中文↔Gloss 平行句對（含 NMS 的格式範本） |
| 教育部常用手語辭典 | 官方詞彙資源 | 擴充 Gloss 詞彙表 [來源 7] |
| 中正大學臺灣手語線上辭典、文化部臺灣手語語料庫 | 詞彙影片／語料庫 | 詞彙查證、語料擴充（**授權範圍需先向網站確認**）[來源 7] |

> 結論：目前真正可直接用於訓練的平行句對僅約 35 句（S01–S30 ＋ P01–P05），遠不足以微調。**資料擴充是本計畫成敗關鍵**，這也是為什麼參考文獻中 ITRI 用「資料合成」、CCL 用「知識庫＋ICL」來補資料不足。

### 3.2 統一格式化（做法依據：CCL 論文 §2.2）

CCL 論文處理了三個標註風格不一的語料庫（詞級標註、含表情標註、手勢級標註），做法是全部轉為統一的 `[src] 中文 => [tgt] Gloss` 格式，表情則作為同步塊附加於手語塊之後 [來源 2]。本計畫比照辦理：

1. 從三批標記表（W、S、P 系列）抽出「中文句 ↔ Gloss 序列 ↔ NMS」三元組。
2. 轉成 JSONL 訓練格式（本計畫決策）：

```json
{"chinese": "你住在宜蘭嗎？", "gloss": "你/宜蘭/住", "nms": "是非問句：眉毛上揚，貫穿全句"}
```

3. 建立《Gloss 詞彙總表》：合併 38 個地名詞彙＋既有句子中出現的所有 Gloss＋教育部辭典查證過的詞條。**這份總表同時就是下游「動作庫檢索」的合法詞彙清單**——模型輸出的每個 Gloss 都必須在表內，否則動作庫找不到對應動作。
   - **2026-07-20 更新（分工確認）**：語言模型端只負責中文→Gloss 的**翻譯語序**；手語影片由下游端自行爬取。因此詞彙總表擴為兩層：第一層自有詞彙（85 Gloss，有自有影片），第二層中正大學《台灣手語線上辭典》詞條（3,500 詞，`data/twtsl/twtsl_words.jsonl`，每詞附辭典示範影片網址供下游爬取）。模型輸出以兩層聯集為合法範圍。

### 3.3 規則驅動的資料合成（做法依據：ITRI ＋ CCL）

ITRI 系統以「合成提示詞生成多樣訓練資料」解決低頻詞問題 [來源 5]；CCL 以人類專家編撰的示例做 in-context learning [來源 2]。本計畫結合兩者：

**步驟 A — 模板合成（規則來自 [來源 7] 的已查證語法）**

以自有的《Gloss 語序決策樹》（已依張榮興 2008、Tai & Tsay 2015、教育部課綱查證，並經母語例句修正）為模板規則：

1. 有情態詞「要」→ `[時間]/[主語]/動詞/[地點或活動]/要`（「要」恆置句尾）
2. 無情態詞、定居類動詞（住/上班）→ `[主語]/地點/動詞`
3. 是非問句 → 不產生「嗎」的 Gloss，NMS 欄位標「眉毛上揚，貫穿全句」
4. 判斷句 → 不產生「是」，直接 `[主語]/[地點]/[身分]`
5. 時間詞（明天、今天…）→ 一律置句首
6. 否定 → 否定詞置動詞後或句尾（Jane Tsay 2021 例：TODAY-I-SCHOOL-GO-NOT）
7. WH 疑問詞 → 置句末，NMS 標註疑問表情（Tai & Tsay 2015）

將《Gloss 詞彙總表》中的地名、人稱、時間詞、動詞代入模板，可系統性生成數百至數千句組合（38 個地名 × 多個句型 × 人稱/時間變化）。

**步驟 B — LLM 輔助改寫中文端**

同一個 Gloss 目標，用 LLM 改寫多樣化的中文說法（「我住台北」「我家在台北」「我目前住在台北」→ 同一 Gloss `我/台北/住`），提升模型對中文表達變化的魯棒性。此即 ITRI「合成提示詞生成多樣訓練資料」的精神 [來源 5]。

**步驟 C — 人工審核（不可省略）**

- 合成句需抽樣送手語老師或聾人顧問審核。依據：資料來源.md 明確警告，未經驗證的規則推廣可能形成錯誤（民俗詞源問題）；且臺灣手語變體無法用單一文獻概括，「若要做教材，應再訪談聾人社群與臺灣手語教師」[來源 7]。SCOPE 的資料集也是由聾人專業人士標註、並以聾人社群問卷驗證系統 [來源 3]。
- 審核通過率、修正紀錄要留存，作為專題報告的資料品質證據。

### 3.4 資料切分（本計畫決策，依一般機器翻譯慣例）

- train / dev / test ≈ 8 : 1 : 1，**以「句型＋詞彙組合」為單位切分**，確保：
  - test 集包含訓練中沒見過的「詞彙 × 句型」組合（測泛化，對應 CCL 對「訓練集樣本 vs 測試集樣本」分開評估的做法 [來源 2]）；
  - 同一句的中文改寫變體不可跨集（避免資料洩漏）。

## 4. 第二階段：模型選擇與環境（預估 1 週）

### 4.1 Gemma 4 規格（已查證）

Gemma 4 由 Google 於 2026-03-31 發布核心版本，Apache 2.0 授權，支援 140 種語言、最長 256K context [來源 6]：

| 變體 | 定位（官方說明） |
|---|---|
| E2B / E4B | 行動與邊緣裝置，低延遲 |
| 12B Unified（2026-06-03 加入） | 無編碼器統一多模態（影像＋音訊） |
| 26B A4B（MoE） | 消費級 GPU |
| 31B Dense | 工作站 |

可從 Hugging Face、Kaggle、Ollama 取得；官方文件提供「Tune using Hugging Face Transformers and QLoRA」等微調教學，DeepMind 頁面亦列出 JAX、Keras、Unsloth 等訓練框架 [來源 6]。

### 4.2 本計畫選型（本計畫決策，理由如下）

- **主力：Gemma 4 E4B ＋ QLoRA** — T2G 是純文字任務，不需多模態；E4B 是官方定位可在邊緣/消費級硬體上運作的最小實用尺寸，QLoRA（4-bit 量化＋LoRA）是官方教學支援的微調路徑 [來源 6]。先例支持：SCOPE 用 Q-LoRA 微調 Qwen2 做手語翻譯達到 SOTA [來源 3]；SignAlignLM 用 LoRA 微調 LLaMA3 **8B** 即優於未微調的 70B 提示法 [來源 1]——小模型＋微調勝過大模型＋純提示，適合學生專題的算力條件。
- **上探（若學校有 GPU 資源）：12B 或 26B A4B**，官方定位即消費級 GPU [來源 6]。
- 訓練框架：Hugging Face `transformers` + `peft` + `bitsandbytes`（官方 QLoRA 教學路徑），或 Unsloth（DeepMind 頁面列名的框架，對單卡訓練有優化）[來源 6]。超參數**從官方 QLoRA 教學的預設值出發**再依 dev 集調整，不自行虛構數值。

## 5. 第三階段：訓練流程（預估 4–6 週）

### Stage A：提示法基線（不訓練，先量測）

先用**未微調**的 Gemma 4 建立三種提示法基線，做法依據 SignAlignLM 的提示詞實驗設計 [來源 1] 與 CCL 的三段式提示詞 [來源 2]：

1. **0-shot**：僅任務描述。
2. **規則提示**：任務描述＋臺灣手語語法規則（第 3.3 節的 7 條規則，皆有文獻出處）。
3. **Few-shot ICL**：任務描述＋規則＋人工審核過的示例句對（CCL 的「任務描述＋專家示例＋翻譯任務」格式 [來源 2]）。

> 目的：(a) 取得比較基準，之後才能證明微調有效；(b) 提前驗證資料格式與評估管線。CCL 證明純 ICL（文心4.0）已能做出可用系統 [來源 2]，若微調效果不如 ICL 基線，代表訓練資料有問題。

### Stage B：QLoRA 監督式微調（SFT）

- 訓練資料：第 3 節建好的 JSONL，套用 Gemma 官方 chat template 組成指令格式（instruction＝任務描述＋規則；input＝中文句；output＝Gloss＋NMS）。
- 方法：QLoRA，依官方教學流程執行 [來源 6]。先例：SCOPE（Q-LoRA on Qwen2）[來源 3]、SignAlignLM（LoRA on LLaMA3 8B）[來源 1]。
- 以 dev 集 BLEU 做 early stopping 與 checkpoint 挑選。

### Stage C：多任務混合訓練（防災難性遺忘）

SignAlignLM 的核心實證發現：**只用手語資料微調會遺忘一般口語能力；把手語任務與一般口語任務（他們用 OpenOrca）交錯混訓，可以在不犧牲口語能力的前提下學會手語處理** [來源 1]。

本計畫做法：T2G 資料混入一定比例的公開中文指令資料（例如通用指令跟隨資料集）一起訓練，混合比例在 dev 集上實驗。這對本專題重要，因為模型上線後還要面對使用者各式各樣的中文輸入（含錯字、口語化表達），不能只會背 Gloss 對照。

### Stage D：推理端知識庫／RAG（部署配置）

依據 ITRI（RAG 解低頻詞）[來源 5] 與 CCL（知識庫語料）[來源 2]：

- 推理時檢索《Gloss 詞彙總表》與相似例句，附進 prompt，把輸出約束在動作庫可用詞彙內。
- 對詞彙表外的新詞（新地名、人名），檢索不到就走 fallback 策略（本計畫決策：輸出「指拼／書空」標記或最近義詞，交下游處理——臺灣手語本有借用詞、字形詞機制 [來源 7]）。

## 6. 第四階段：評估（預估 2 週，與訓練交錯進行）

### 6.1 自動評估

- **BLEU、ROUGE**：三篇論文共同採用的標準指標（SignAlignLM：BLEU/ROUGE/BERTScore [來源 1]；SCOPE：BLEU/ROUGE-L [來源 3]；CCL：BLEU/ROUGE [來源 2]）。對 Gloss 序列以 `/` 切詞後計算。
- **詞彙表內率（in-vocabulary rate）**（本計畫自訂指標）：輸出 Gloss 落在《Gloss 詞彙總表》內的比例。理由：下游是動作庫檢索，表外詞等於無法播放，此指標直接反映系統可用性。（2026-07-20 起以兩層詞彙聯集〔自有 85＋twtsl 3,500〕為分母計算；下游影片自行爬取，見 3.2 節更新）

### 6.2 人工評估

- 5 分制人工評分（做法依據 CCL 的人工評測設計，其採 5 分制、多維度 [來源 2]）：語意正確性、語序正確性、NMS 正確性、下游可播放性。
- 邀請手語老師／聾人使用者參與（SCOPE 以聾人社群問卷驗證系統有效性 [來源 3]；資料來源.md 亦強調需聾人社群參與 [來源 7]）。

### 6.3 錯誤分析

依 CCL 的錯誤分類法 [來源 2] 統計：Gloss 缺失、Gloss 多增、語序錯誤，另加 NMS 錯誤（本計畫擴充）。分析結果回饋到 3.3 節的資料合成規則，形成迭代。

### 6.4 對照組

| 對照 | 目的 |
|---|---|
| Stage A 提示法 vs Stage B 微調 | 證明微調的價值（SignAlignLM 的核心比較 [來源 1]） |
| 純 T2G 微調 vs 多任務混訓 | 驗證防遺忘效果（用一般中文問答測試集測退化程度 [來源 1]） |
| 有 RAG vs 無 RAG | 驗證低頻詞改善（ITRI 的主張 [來源 5]） |

| 8–10 | Stage C 多任務混訓實驗 |
| 10–11 | Stage D RAG 整合；與下游動作庫端串接測試 |
| 11–13 | 人工評估、錯誤分析、消融實驗、撰寫報告 |

## 8. 風險與限制（皆有文獻依據）

1. **Gloss 表示法的固有缺陷**：資訊丟失（非手動通道、3D 空間）、跨語料標註不一致、標註需專家、聾人日常不使用 Gloss（CCL 引 Müller 2023 [來源 2]）。緩解：NMS 獨立欄位＋統一標註規範＋聾人審核。
2. **合成資料品質**：規則模板可能過度簡化真實手語；臺灣手語有地區與年齡變體，單一文獻無法概括 [來源 7]。緩解：3.3-C 人工審核、6.2 聾人評估。
3. **資料量**：即使合成後，規模仍遠小於 PHOENIX-14T、CSL-Daily 等基準資料集；句型多樣性受模板限制。緩解：明確界定初版系統的領域範圍（如：地名、自我介紹、日常問候），對外聲明能力邊界。
4. **外部語料授權**：文化部臺灣手語語料庫、中正大學辭典的資料使用授權需先確認 [來源 7]，未確認前不得抓取入訓練集。
5. **模型幻覺**：LLM 可能輸出詞彙表外的 Gloss。緩解：Stage D 的 RAG 約束＋詞彙表內率監控。
